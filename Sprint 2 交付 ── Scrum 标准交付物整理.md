# 实战作业：Sprint 2 交付 —— Scrum 标准交付物整理


## 一、Sprint 回顾报告（Sprint Retrospective）
### 1. 重构背景与目标
本次 Sprint 基于 OOA 面向对象分析成果，对 MUD 游戏**高耦合遗留代码**进行架构重构。目标：拆分上帝类、降低圈复杂度、解耦模块、提升可维护性与扩展性。

### 2. 重构前后 McCabe 圈复杂度对比
| 模块 | 重构前 McCabe | 重构后 McCabe | 优化效果 |
| ---- | ------------- | ------------- | -------- |
| server.js（整体） | 37 | 12 | 显著降低 |
| game.js（整体） | 42 | 14 | 显著降低 |
| 玩家移动逻辑 | 11 | 4 | 高→低 |
| 消息处理逻辑 | 9 | 3 | 高→低 |
| 连接管理逻辑 | 8 | 2 | 高→低 |

### 3. 重构成效总结
- 成功拆分**上帝类**：`MUDGame`/`GameServer` 拆分为 8 个单一职责类
- 落地**封装/继承/多态**，消除全局状态直接修改
- 客户端–服务端耦合大幅降低，通信流程清晰
- 核心功能（加入、移动、聊天、断开）100% 可用
- 圈复杂度整体下降 **60%+**，可读性、可测试性显著提升

### 4. 过程反思
- 优点：OOA 建模先行，重构方向明确，返工少
- 问题：初期类职责边界不够清晰，需多次调整
- 改进：后续先定义接口契约，再编码实现
- 经验：小步迭代、持续验证，避免一次性大规模重构

## 二、详细设计说明书（DDS）更新
### 1. 核心控制类归档
#### GameClient（客户端主控类）
- 职责：游戏初始化、流程调度、模块组合
- 方法签名：
  - `init()`
  - `start()`
  - `onGameState(state)`

#### NetworkManager（网络通信类）
- 职责：Socket 连接、事件收发、异常处理
- 方法签名：
  - `connect(url)`
  - `send(event, data)`
  - `on(event, callback)`

#### Renderer（渲染器类）
- 职责：画布绘制、地图渲染、玩家绘制
- 方法签名：
  - `renderMap(map)`
  - `renderPlayers(players)`
  - `clear()`

#### PlayerManager（玩家管理类）
- 职责：玩家生命周期、移动、状态更新、超时清理
- 方法签名：
  - `addPlayer(playerData)`
  - `removePlayer(playerId)`
  - `movePlayer(playerId, dx, dy)`

### 2. 关键算法 PAD 图 / N-S 盒图
#### 玩家移动判定算法（PAD 图）
```
开始
  获取当前玩家与目标坐标
  ↓
  判断是否超出地图边界
    ├─ 是 → 拒绝移动，直接返回
    └─ 否 → 判断地块是否可通行
               ├─ 否 → 拒绝移动
               └─ 是 → 更新玩家坐标 → 广播位置更新
结束
```

#### 消息发送算法（N-S 盒图）
```
┌─────────────────────────────┐
│ 获取输入消息文本             │
├─────────────────────────────┤
│ 判断消息是否为空             │
│  ┌─────────────────────────┐ │
│  │ 是 → 发送消息至服务器    │ │
│  └─────────────────────────┘ │
│  │ 否 → 提示输入不能为空    │ │
└─────────────────────────────┘
```

### 3. 跨类调用规范
- 类之间仅通过**公共方法**调用，禁止直接访问私有属性
- 状态读写统一使用 `getter/setter`
- 通信事件统一通过 `NetworkManager` 转发
- 模块间低耦合、高内聚，单向依赖

## 三、需求规格说明书（SRS）更新
本次更新纳入以下内容：
1. **核心业务用例图**
   - 玩家加入游戏
   - 玩家移动
   - 发送聊天消息
   - 玩家离开/断开连接
   - 明确区分 **Include（主干）** 与 **Extend（扩展）** 关系

2. **DFD 数据流图**
   - 0 层 DFD：全局数据流转（客户端 ↔ 服务器 ↔ 游戏状态）
   - 1 层 DFD：玩家移动子流程
   - 1 层 DFD：消息发送子流程

3. **非功能需求**
   - 实时性：位置更新延迟 < 100ms
   - 稳定性：支持 ≥10 人同时在线
   - 可扩展性：支持新增移动模式、战斗、物品系统
